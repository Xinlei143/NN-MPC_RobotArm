## 3. 两种Projection不能混为一谈

### 3.1 Planner projection

planner projection发生在CEM population rollout内部：

```text
candidate residual
    -> raw IK nominal + residual
    -> velocity/acceleration/joint-limit projection
    -> GRU rollout
    -> tracking and actuator cost
```

它的目标是让GRU rollout和cost评估更接近实际可执行命令。

### 3.2 Execution projection

execution projection发生在100 Hz控制层：

```text
packet MPC residual
    + bounded feedback
    -> requested absolute command
    -> braking-aware physical projection
    -> actual actuator command
```

execution projection是安全约束，必须始终开启。它必须覆盖：

- 正常active packet；
- 启动时还没有首包；
- packet expiration；
- planner failure；
- late packet drop；
- requested correction恰好为零；
- residual与feedback相互抵消。

因此：

```text
planner_projection=off
```

不代表关闭安全约束，只表示CEM内部不对每条population candidate运行完整physical
projection。100 Hz执行层仍必须严格投影。

## 4. 已确认并修复的执行层Bug

旧执行投影对零correction有特殊分支：直接返回raw IK nominal，不执行速度和加速度
限制。

当replacement packet缺失时，故障链为：

```text
planner failure / late drop / solve不及时
    -> 没有replacement packet
    -> 旧packet继续执行到horizon末端
    -> packet age从H-1变成-1
    -> requested correction变成0
    -> zero-correction分支直接返回nominal
    -> command从旧residual command突然跳回nominal
    -> 出现约1000--1500 rad/s²的离散加速度spike
```

planner failure不会立即发布zero packet。它只是不发布新packet；真正的跳变发生在旧
packet随后耗尽时。

已经完成的修复：

- 删除zero-correction bypass；
- 零与非零correction使用相同执行投影；
- 统一NumPy/Torch的velocity、acceleration、joint-limit和braking envelope；
- packet gap使用constraint-projected Direct-IK fallback；
- 区分普通planner failure、late drop、packet expiration和worker fatal；
- 每个planner result使用唯一ID，避免逐tick重复统计；
- 保存failure reason和event log。

修复后诊断结果：

```text
maximum command acceleration:       12.5003 rad/s²
command acceleration violations:    0
control deadline misses:            0
packet expirations:                 0
```

这些修复与planner projection开关无关，应该保留。

## 5. 为定位问题加入的语义拆分

后续修改将过去含义混杂的`executed_residual`拆成：

```text
requested_mpc_residual
requested_feedback_correction
requested_total_correction
requested_absolute_command
safety_projection_offset
command_nominal_offset
planner_execution_qref_error
```

定义为：

```text
requested_total_correction
    = clip(requested_mpc_residual + requested_feedback_correction)

requested_absolute_command
    = nominal + requested_total_correction

safety_projection_offset
    = actual_command - requested_absolute_command

planner_execution_qref_error
    = actual_command - planner_rollout_qref[packet_age]
```

packet也同时保存：

- CEM请求的residual sequence；
- planner rollout实际使用的q-ref sequence；
- predicted state sequence。

这些日志拆分是必要的，因为它们分别回答：

```text
CEM请求了什么？
fast feedback增加了什么？
safety filter修改了什么？
机器人最终执行了什么？
```

## 7. Planner Projection On/Off的现有诊断

固定条件：旧GRU、正式circle reference、D7、H20、128 samples、2 iterations、
100 Hz执行层和相同feedback gains。

诊断表使用相同`projected_offset` cost来隔离projection开关。它不是最终
`requested` residual-cost实验。

| Planner projection | Seed | Virtual full/lap | Threaded full/lap | Threaded E2E p95 | Worker rate |
|---|---:|---:|---:|---:|---:|
| on | 0 | 36.62 / 39.91 mm | 41.15 / 43.99 mm | 62.50 ms | 19.08 Hz |
| on | 1 | 36.49 / 39.53 mm | 44.95 / 49.78 mm | 63.58 ms | 18.77 Hz |
| off | 0 | 37.07 / 37.49 mm | 37.61 / 39.83 mm | 48.72 ms | 25.82 Hz |
| off | 1 | 44.04 / 49.03 mm | 41.92 / 45.85 mm | 48.36 ms | 25.94 Hz |

在`off`下：

```text
seed 0 threaded - virtual: full +0.54 mm, lap +2.35 mm
seed 1 threaded - virtual: full -2.13 mm, lap -3.17 mm
```

结果说明：

1. planner projection的当前实现确实增加约14--16 ms E2E latency；
2. D7和5 ms guard时，publication deadline约65 ms，`on`只剩约1--3 ms余量；
3. seed 0中virtual基本不变，而threaded从41.15改善到37.61 mm，说明worker变快有
   真实收益；
4. seed 1中virtual `off`自身从36.49退化到44.04 mm，因此不能把所有差距缩小都
   解释为threaded被修好；
5. 两个seed不足以冻结正式配置。

## 8. Planner Projection真的有必要吗

完整per-candidate planner projection不是方法成立的必要条件。

必须存在的是：

```text
机器人实际执行的命令始终满足物理约束
```

并不要求必须用当前昂贵实现，对CEM每条候选序列执行完整递推projection。

planner projection `on`的理论优点：

- GRU rollout看到可执行命令；
- tracking/torque cost基于可执行命令；
- prediction--execution mismatch可能更小。

当前实现的实际缺点：

- H20逐步递推产生大量小CUDA kernel；
- E2E p95从约48 ms增加到约63 ms；
- worker rate从约26 Hz下降到约19 Hz；
- 更容易接近或超过packet deadline；
- projection改变cost landscape，旧cost weights不再适用；
- raw IK nominal自身不可立即执行时，projected offset可能不再代表CEM action；
- 不同residual cost定义会得到明显不同tracking结果。

所以当前结论是：

> execution projection必须保留；完整planner population projection不应继续作为默认
> paper配置。它可以保留为算法一致性消融或未来优化目标。

## 9. 为什么不能简单长期使用Planner Projection Off

`off`虽然实时性能更好，但execution safety filter修改planner动作的频率很高：

```text
virtual activation rate:  84.1% -- 87.5%
threaded activation rate: 85.8% -- 87.0%
```

全轨迹safety projection offset RMS：

```text
virtual seed 0:  15.68 mrad
virtual seed 1:  23.15 mrad
threaded seed 0:  9.14 mrad
threaded seed 1: 15.95 mrad
```

这不是数值噪声，说明planner经常请求不能原样执行的command。论文如果直接使用
`off`，必须诚实描述execution filter，并报告projection discrepancy和
planner-execution qref error。

更重要的是，应当让planner用低成本方式避免明显不可执行的候选，而不是完全忽略
constraints后全部交给execution filter。

## 10. 两个Seed为什么差异明显

当前实验中的plant和reference基本确定，seed主要控制CEM sampling，因此seed方差
表示optimizer stochasticity，不是环境随机性。

逐lap结果：

| Mode | Seed | Lap 0 | Lap 1 | Lap 2 |
|---|---:|---:|---:|---:|
| Virtual off | 0 | 36.70 | 37.21 | 38.53 mm |
| Virtual off | 1 | 38.97 | 52.23 | 54.44 mm |
| Threaded off | 0 | 39.32 | 39.53 | 40.63 mm |
| Threaded off | 1 | 40.19 | 55.16 | 40.62 mm |

seed 1主要从第二圈开始进入更差的闭环路径。

Lap 1的projection discrepancy：

```text
virtual seed 0:   9.35 mrad
virtual seed 1:  33.31 mrad
threaded seed 0:  8.91 mrad
threaded seed 1: 27.04 mrad
```

对应torque RMS也明显增加，但requested residual RMS变化较小，且没有residual
saturation。这说明seed 1不是简单输出更大的residual，而是输出了更容易被有状态
velocity/acceleration filter修改的时序。

当前CEM只有128 samples和2 iterations，并且warm-start mean启用、
`reset_std_each_step=False`。一次不同elite selection会改变后续mean/std；状态偏离后，
下一次CEM又在不同状态上优化，差异通过闭环逐步放大。

同一seed也不能让virtual和threaded逐次使用相同候选：

- virtual固定每5步规划，约362次solve；
- threaded strict ASAP，约467--469次solve；
- 两者随机数消费次数、state、history和warm-start shift都不同。

所以论文中应表述为`five paired CEM seeds`，不能说是五个独立随机环境trial，也不能
将paired seed解释为逐action的common random numbers。

## 11. 推荐替代方案：向量化约束检测与惩罚

推荐不再对每条population candidate运行完整递推physical projection，而是对请求
序列执行向量化约束计算。

对于：

```text
q_req[t] = q_nom[t] + r[t]
```

一次张量操作计算：

```text
v[0] = (q_req[0] - previous_command) / dt
v[t] = (q_req[t] - q_req[t-1]) / dt

a[0] = (v[0] - previous_velocity) / dt
a[t] = (v[t] - v[t-1]) / dt
```

加入constraint cost：

```text
C_constraint =
    lambda_v * relu(abs(v) - v_max)^2
  + lambda_a * relu(abs(a) - a_max)^2
```

实现策略：

- 轻微超限使用soft penalty；
- 明显超限候选标记为infeasible；
- 始终插入一条已知可行baseline；
- 有效候选过少时缩小CEM std后重新采样；
- execution physical projection仍作为最后安全网；
- 日志继续报告实际filter intervention。

优点：

- velocity/acceleration difference完全向量化；
- 避免H个连续的小CUDA kernel；
- planner不会完全忽略可执行性；
- 不改变residual MPC和current-nominal reanchoring主线；
- 有机会降低projection activation和seed variance；
- 保留100 Hz执行层安全保证。

另一个更彻底的方法是直接优化command acceleration，然后积分得到velocity和
position。该方法可以by construction满足约束，但会改变MPC action space和论文方法，
暂时不建议采用。

## 15. 当前最终判断

本轮问题不是单一bug，而是三件事叠加：

```text
paper commit静默改变planner projection默认语义
+ packet gap时zero-correction绕过执行约束
+ CEM stochasticity与有状态execution filter相互放大
```

必须保留执行安全bug修复，但没有必要保留当前昂贵的完整planner population
projection作为论文主配置。

推荐最终结构：

```text
planner:
    requested residual action
    vectorized velocity/acceleration feasibility and penalty
    no full per-candidate recursive physical projection

execution:
    bounded MPC residual + bounded fast feedback
    shared braking-aware physical projection always on
    explicit projection discrepancy logging
```

在这一结构完成并通过小规模定位实验前，不应继续跑完整论文实验，也不应使用当前
两个seed决定论文最终projection配置。
