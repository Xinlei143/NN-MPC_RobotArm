# NN-MPC_RobotArm 项目计划书

> **历史计划与当前实现边界**：本文保留早期 `delta_q_ref` / reference-increment 路线、实验矩阵和论文计划。当前默认实现已经改为 residual MPC：`q_ref=q_nom+r`，其中 `q_nom` 是可执行 IK nominal，`r` 为有界补偿；默认选择策略是 `lowest_cost`。需要描述当前代码时，以 [README](../README.md)、[CostFunction](260715_CostFunction.md) 和 [MPC 伪代码](260715_mpc-pseudocode.md) 为准；本文中旧动作变量仅表示历史方案或消融候选。

本文档基于当前仓库代码、`docs/260716_项目发展思路.md`、`docs/260719_PROJECT_STRUCTURE.md`、`docs/260722_PROJECT_COMPLETION_STATUS.md` 和 `docs/260715_mpc-pseudocode.md` 整理，目标是把项目从当前的 learned dynamics + joint-space CEM-MPC 雏形推进到一条可写论文、可复现实验、可解释轨迹追踪能力边界的完整路线。

当前项目的合理定位不是泛泛地做“神经网络机械臂 MPC”，而是研究：

> 在只开放 position actuator reference 的工业机械臂中，如何学习 `q_ref -> q, dq` 的闭环 actuator-reference dynamics，并用采样式 CEM-MPC 优化可执行的 actuator reference，使实际机械臂更稳定地跟踪期望轨迹，同时处理 MPC 闭环运行带来的数据分布偏移。

## 1. 对 `260716_项目发展思路.md` 的评价

### 1.1 总体判断

`docs/260716_项目发展思路.md` 的总体方向是合理的，而且和当前代码主线基本一致。它最有价值的地方是把项目问题从“用神经网络预测动力学”收窄为更具体的 position-control 闭环建模问题。

当前代码已经支持这条叙事：

- `dynamics_modeling/neural_dynamics/mujoco_env.py` 中环境状态是 `[q, dq]`，动作是 absolute actuator target joint angles，也就是 `q_ref`。
- `mpc/reference.py` 生成期望关节轨迹 `q_des` 和有限差分速度 `dq_des`。
- 当前 `mpc/planner_rollout.py` 将 CEM 的 normalized residual 加到 projected nominal `q_nom`，再投影为可执行的 absolute `q_ref_sequences`；后文的 `delta_q_ref` 描述属于历史方案。
- `dynamics_modeling/neural_dynamics/rollout.py` 用 learned dynamics 在候选 `q_ref_sequences` 下预测未来 `[q, dq]`。
- `mpc/cost_functions.py` 用关节位置误差、速度误差、控制平滑、terminal error 和 joint-limit penalty 给候选轨迹评分。
- `scripts/collect_mpc_data.py`、`scripts/model_c/evaluate.py`、`scripts/analyze_ood_mpc.py` 已经形成 Model A/B/C 与 OOD 分析的工程雏形。

因此，文档中提出的三条贡献方向成立：

1. actuator-reference-aware closed-loop dynamics learning；
2. CEM-MPC over actuator reference increments；
3. MPC-induced data aggregation and OOD diagnostics。

### 1.2 需要修正或补强的地方

当前发展思路仍有几个风险点，需要在后续项目中显式处理。

第一，当前“轨迹追踪”实际上是 joint-space reference tracking。`mpc/reference.py` 只支持 `hold`、`step`、`joint_sine`、`multi_joint_sine` 四类关节空间参考；`mpc/cost_functions.py` 只计算 `q_pred - q_des` 和 `dq_pred - dq_des`，没有任务空间末端位置或位姿误差。因此论文中不能直接声称已经实现了 end-effector trajectory tracking。

第二，`scripts/run_cem_mpc.py` 里的 `predicted_real_error_gap = best_cost - realized_error` 量纲不一致：`best_cost` 是加权综合 cost，`realized_error` 是单步关节误差范数。这个字段可以作为临时诊断，但不能作为论文主指标。后续必须新增同量纲预测误差，例如 `||predicted_next_q - real_next_q||`、`||predicted_rollout_q - real_rollout_q||`。

第三，`260716_项目发展思路.md` 中相关工作链接需要重新核验。最终论文引用必须基于实际阅读过的论文和稳定 bibliographic metadata，不能直接沿用带 `utm_source` 的临时链接。

第四，当前项目有 pipeline，但还缺实验闭环证据。若没有 direct baseline、filtered baseline、Model A/B/C 闭环对比、OOD 下降证据和扰动实验，项目会退化成普通 NN-MPC 仿真演示。

## 2. 当前项目状态

### 2.1 已完成能力

项目已经具备以下基础能力：

- ABB IRB 2400 MuJoCo 仿真环境；
- position actuator control interface；
- 状态 `[q, dq]` 与动作 `q_ref` 的数据采集；
- MLP、GRU、Transformer 三类 learned dynamics；
- normalizer、dataset、history window、rollout loss 和 learned rollout 工具；
- CEM-MPC 闭环控制主线；
- `delta_q_ref -> absolute q_ref` 的 actuator reference 规划；
- joint limit、rate limit、smoothness penalty 和 terminal tracking cost；
- MPC rollout 保存、CSV 日志和基础图像；
- MPC-induced dataset 收集；
- Model A/B/C checkpoint 对比脚本；
- state/action z-score OOD 分析脚本；
- 顶层 MPC pipeline 单元测试。

### 2.2 当前主要缺口

后续项目推进时，优先补齐以下缺口：

1. 基线实验不完整：需要 direct `q_ref=q_des`、filtered reference、现有 CEM-MPC 的统一评估。
2. 预测误差指标不够严谨：需要同量纲 one-step、multi-step、closed-loop prediction-real gap。
3. 任务空间轨迹追踪缺失：目前只有关节空间命令追踪，需要加入 end-effector reference、IK/FK 和末端误差记录。
4. OOD 分析仍偏粗：当前只按整体 state/action z-score 统计，需要补充 per-joint、per-trajectory、per-controller 的分布偏移。
5. 鲁棒性实验缺失：需要高频轨迹、限位压力、初值扰动、payload 或 actuator mismatch。
6. 论文证据链未闭合：需要把 prediction error、closed-loop tracking、OOD fraction 和 failure rate 放到同一组实验中解释。

## 3. 项目目标与验收标准

### 3.1 工程目标

构建一个可复现的 position-controlled manipulator learned CEM-MPC 实验平台，使项目能够稳定完成：

1. 训练 learned dynamics；
2. 运行 direct/filtered/CEM-MPC baselines；
3. 收集 MPC-induced data；
4. 训练和比较 Model A/B/C；
5. 分析 OOD 与闭环性能；
6. 支持 joint-space 和 task-space position trajectory tracking 两类参考；
7. 输出论文级图表和指标表。

### 3.2 研究目标

最终论文应回答四个问题：

1. 优化 actuator reference `q_ref` 是否比直接发送 `q_des` 更好？
2. learned dynamics 的 one-step loss、多步 rollout error 和 closed-loop MPC 性能之间是什么关系？
3. MPC-induced data aggregation 是否能降低 OOD fraction 并提升闭环控制？
4. 在高频轨迹、限位附近、初值扰动或模型 mismatch 下，方法是否比 baseline 更稳定？

### 3.3 验收指标

每组控制实验至少报告：

- joint tracking RMSE；
- joint tracking max error；
- end-effector position RMSE；
- end-effector position max error；
- failure rate；
- joint-limit violation count；
- mean planning time；
- p95 planning time；
- mean `||q_ref - q||`；
- mean `||delta_q_ref||`；
- mean one-step prediction error；
- multi-step rollout prediction error；
- state OOD fraction；
- action OOD fraction。

其中 joint tracking 和 end-effector tracking 应同时报告。即使任务空间轨迹由 IK 转成 `q_des` 后再控制，也要用实际 MuJoCo 末端位置 `p_ee` 和目标 `p_des` 计算最终任务空间误差。

## 4. 总体技术路线

项目分为五个阶段推进。

### 阶段一：固化当前 joint-space CEM-MPC 主线

目标是让当前关节空间 pipeline 成为可靠基线。

主要工作：

- 固定一个可用 checkpoint、normalizer、reference mode 和随机种子集合；
- 用同一套初值和 reference 运行 direct、filtered、learned CEM-MPC；
- 把 `q_des`、direct `q_ref=q_des`、CEM-selected `q_ref`、actual `q` 画在同一张图里；
- 新增严格的 prediction-real gap 指标；
- 保存所有实验配置到输出目录，保证结果可复现。

建议新增脚本或参数：

- `scripts/run_direct_reference.py`：直接发送 `q_ref=q_des`；
- 或在 `scripts/run_cem_mpc.py` 中新增 `--controller_mode direct|filtered|cem`；
- `--filter_alpha` 用于 filtered reference baseline；
- `--save_config` 默认开启，保存 args、checkpoint path、normalizer path、git commit hash。

阶段验收：

- direct、filtered、CEM 三个 baseline 能在同一 reference 下运行；
- 每次 rollout 都输出 joint tracking 图、control 图、error CSV 和 config；
- 顶层测试通过。

### 阶段二：完成 Model A/B/C 数据闭环

目标是证明 MPC-induced data aggregation 对闭环控制有意义。

模型定义：

- Model A：用普通结构化离线数据训练，例如 hold、step、smooth waypoint、sine 的混合 `q_ref` 数据；
- Model B：用更接近 MPC 查询分布的数据训练，例如更高频、更小步长、更多 rate-limited `q_ref` 序列；
- Model C：用 Model A/B 闭环运行 CEM-MPC 后收集的 MPC-induced data 追加或重采样训练。

主要工作：

- 固化每个模型对应的数据集路径、checkpoint 路径和 normalizer 路径；
- 使用 `scripts/model_c/evaluate.py` 在完全相同控制配置下评估 A/B/C；
- 使用 `scripts/analyze_ood_mpc.py` 分析每个模型 rollout 相对自身训练集和 baseline dataset 的 OOD fraction；
- 补充预测误差评估：在 MPC rollout 的真实 `state_t, q_ref_t, state_{t+1}` 上评估 one-step error；
- 补充 multi-step open-loop replay：从真实 rollout 中取窗口，用真实未来 `q_ref` rollout learned model，比较预测状态和真实状态。

阶段验收：

- 产生 `model_abc_summary.csv`；
- 产生 `ood_summary.csv`；
- 产生 one-step 和 multi-step prediction error 表；
- 能回答 Model C 是否在 closed-loop tracking、OOD fraction 或 failure rate 上优于 A/B。

### 阶段三：从关节空间命令追踪扩展到任务空间轨迹追踪

这是当前项目最需要补强的地方。

#### 4.3.1 当前问题

当前轨迹追踪链路是：

```text
reference generator -> q_des(t), dq_des(t)
CEM samples delta_q_ref
delta_q_ref -> q_ref sequence
learned dynamics predicts q(t), dq(t)
cost = joint-space error between q_pred and q_des
MuJoCo executes selected q_ref
```

这只能说明项目能跟踪关节空间参考，不能说明它已经能直接跟踪末端轨迹。即使末端实际运动跟着变化，也只是关节轨迹的结果，不是任务空间目标进入了控制问题。

#### 4.3.2 推荐方案：任务空间参考先经 IK 转成关节参考

第一阶段不要把控制输入改成任务空间命令，也不要重训 dynamics。推荐路线是：

```text
task-space trajectory p_des(t)
        |
        v
IK / constrained inverse kinematics
        |
        v
smooth feasible q_des(t), dq_des(t)
        |
        v
existing learned CEM-MPC optimizes delta_q_ref
        |
        v
MuJoCo position actuator executes q_ref
        |
        v
evaluate both joint error and end-effector error
```

这样做的理由：

- 当前 learned dynamics 已经围绕 `[q, dq, q_ref] -> next [q, dq]` 建好，不需要改数据 schema；
- position actuator interface 的真实输入仍然是 `q_ref`，符合项目核心问题；
- IK 失败、限位、平滑性可以在参考生成层处理，不污染 CEM 主逻辑；
- 可以快速得到 end-effector trajectory tracking 指标；
- 后续如果要做 task-space cost，可以在已有 joint-space 版本稳定后逐步加入。

#### 4.3.3 具体实现建议

新增 `mpc/kinematics.py`：

- `forward_kinematics(model_xml, q) -> p_ee`；
- `batch_forward_kinematics(model_xml, q_sequence) -> p_ee_sequence`；
- `solve_ik_position(model_xml, p_target, q_seed, joint_low, joint_high, q_nominal, max_iters, tol) -> IKResult`；
- `IKResult` 字段包括 `q`、`success`、`position_error`、`num_iters`、`reason`。

新增或扩展 `mpc/reference.py`：

- `generate_task_space_reference(mode, initial_ee_position, total_steps, control_dt, amplitude, seed)`；
- 支持 `line`、`circle`、`figure8`、`hold`；
- 输出 `p_des` 和 `dp_des`；
- `task_reference_to_joint_reference(...)` 调用 IK，把 `p_des` 转成 `q_des`。

IK 目标函数建议为：

```text
min_q ||FK(q) - p_des||^2
    + lambda_smooth  * ||q - q_prev||^2
    + lambda_posture * ||q - q_nominal||^2
```

约束：

- `joint_low + margin <= q <= joint_high - margin`；
- 单步 `||q_t - q_{t-1}||_inf <= q_des_rate_limit`；
- IK 失败时保持上一可行 `q_des`，并记录 `ik_failure_flag=1`；
- 若连续 IK 失败超过阈值，当前任务空间轨迹标记为 infeasible，不进入论文主实验。

`scripts/run_cem_mpc.py` 新增参数：

- `--tracking_space joint|task_position`；
- `--task_reference_mode hold|line|circle|figure8`；
- `--task_reference_amplitude`；
- `--ik_max_iters`；
- `--ik_tol`；
- `--ik_smooth_weight`；
- `--ik_posture_weight`。

输出新增字段：

- `ee_position`；
- `ee_desired_position`；
- `ee_tracking_error`；
- `ik_failure_flags`；
- `ik_position_error`；
- `task_reference_mode`。

`mpc/logging.py` 新增图：

- `ee_tracking.png`：x/y/z 维度的 `p_ee` vs `p_des`；
- `ee_tracking_error.png`：`||p_ee - p_des||`；
- `ee_path_xy.png`：XY 平面路径；
- `ee_path_3d.png`：三维路径。

#### 4.3.4 后续增强：直接任务空间 cost

IK 方案稳定后，可以加入任务空间 cost：

```text
cost = w_q * joint_tracking
     + w_ee * ||FK(q_pred) - p_des||^2
     + w_dq * velocity_cost
     + w_u * actuator_offset
     + w_du * smoothness
     + w_limit * joint_limit
```

注意事项：

- CEM 每次会评估大量候选轨迹，直接对所有 `q_pred` 调用 MuJoCo FK 可能太慢；
- 应优先实现轻量 batched FK，或者只对 elite / final candidates 计算任务空间 cost；
- 在没有高效 FK 前，不建议把 `w_ee` 作为主控制目标；
- V1 任务空间追踪以 IK-generated `q_des` 为主，V2 再加入 direct task-space cost。

阶段验收：

- `--tracking_space task_position` 能运行 line/circle/figure8；
- rollout 中同时保存 joint 和 end-effector 指标；
- 任务空间轨迹下，实际 `p_ee` 与 `p_des` 的 RMSE 可统计、可画图；
- 即使 IK 失败，也有清晰日志，不会静默生成错误参考。

### 阶段四：鲁棒性与约束实验

目标是证明方法不是只在单一 sine trajectory 上有效。

实验组：

1. 高频 reference sweep；
2. reference amplitude sweep；
3. joint-limit stress test；
4. initial-state perturbation；
5. payload perturbation；
6. actuator gain mismatch；
7. limited CEM budget test。

每组比较：

- direct reference；
- filtered reference；
- learned CEM-MPC Model A；
- learned CEM-MPC Model C；
- 可选 learned proposal CEM。

阶段验收：

- 每组实验有统一 summary CSV；
- 每组实验至少 3 个 seed；
- 每组实验记录 failure reason；
- 能区分 tracking error 上升是来自 OOD、预测误差、限位还是 CEM budget 不足。

### 阶段五：论文与增强模块

主线实验稳定后，再考虑增强项。

可选增强：

- learned CEM warm start；
- proposal network；
- ensemble uncertainty；
- task-space cost；
- Koopman 或 simple local linear dynamics baseline；
- real-time profiling 和 sample budget trade-off。

这些增强不能替代主线贡献。若时间有限，优先保证 Model A/B/C + OOD + task-position tracking 的证据链完整。

## 7. 论文结构建议

### 7.1 标题方向

推荐题目：

> Actuator-Reference-Aware Learned CEM-MPC for Position-Controlled Manipulators

备选题目：

> Closed-Loop Dynamics Learning and MPC-Induced Data Aggregation for Position-Controlled Robot Arms

### 7.2 方法章节

Method 建议分为：

1. Problem Formulation；
2. Actuator-Reference-Aware Dynamics Learning；
3. CEM-MPC over Reference Increments；
4. Joint-Space and Task-Space Reference Tracking；
5. MPC-Induced Data Aggregation；

## 8. 风险与应对

### 风险 1：任务空间 IK 不稳定

应对：

- V1 只做末端位置，不做姿态；
- 轨迹幅度从小开始；
- 每一步以上一时刻 IK 解为 seed；
- 加 smoothness 和 posture regularization；
- 失败时保持上一可行解并记录 failure；
- 不把 IK 失败轨迹纳入主实验。

### 风险 2：CEM 计算太慢

应对：

- 保持主 cost 为 joint-space cost；
- task-space cost 先只用于 logging；
- 若加入 `w_ee`，先只对 elite 或 final candidates 计算；
- 对 sample budget 做 128/256/512/1024 sweep；
- 报告 planning time，而不是只报告 tracking error。

### 风险 3：Model C 不一定全面优于 Model A

应对：

- 不只看 one-step validation loss；
- 报告 closed-loop tracking、OOD fraction 和 MPC rollout replay error；
- 如果 Model C 只在 MPC distribution 上更好，也符合项目论点；
- 若 Model C 过拟合，调整 MPC-induced data ratio。

### 风险 4：论文贡献不够新

应对：

- 避免声称 CEM + learned dynamics 本身新；
- 强调 position-control actuator-reference interface；
- 强调 `q_des`、`q_ref`、actual `q` 的闭环差异；
- 强调 MPC-induced distribution shift；
- 加入任务空间轨迹追踪时，也保持“可执行 actuator reference 优化”这个主线。

## 10. 最终结论

`260716_项目发展思路.md` 的研究定位是合理的，但必须把“当前只做关节空间命令追踪”这个事实讲清楚。项目下一步不应该直接把控制输入改成任务空间命令，而应先通过任务空间轨迹生成和 IK，把 `p_des(t)` 转成平滑可行的 `q_des(t)`，再沿用当前 learned CEM-MPC 优化 actuator reference `q_ref`。

这样做既保留了项目最有特色的 actuator-reference-aware closed-loop dynamics 主线，也能补上审稿人会关心的 end-effector trajectory tracking 证据。最终项目应以“position-control interface 下的 actuator reference 优化 + MPC-induced data aggregation + joint/task-space tracking 评估”作为完整贡献闭环。
